Software Development Models

Understanding how SDLC phases are arranged to fit different projects

In the previous session, we explored the Software Development Life Cycle (SDLC) and how AI is transforming each of its 7 phases. Now, let's shift focus to Software Development Models — the different ways the SDLC phases can be arranged to fit different types of projects. We'll look at three of the most commonly used models: the Waterfall Model, the Agile Model, and the V-Model.

Why Not One SDLC for All?

Every project differs in size, requirements, and flexibility — one approach doesn't fit all.

Types of Software Projects

DimensionRange
SizeSmall → Big
RequirementsFixed → Always Changing
FlexibilityLow → High

For Example

Example 1 — Simple Task App
  • Small, simple project with just a few basic features
  • Requirements are known from the start
  • Once built, it doesn't change much
  • Example: A basic to-do list app
Example 2 — Instagram
  • Big, complex project with many features
  • Requirements constantly change based on user feedback
  • Always adding new features and updates
  • Began as photo sharing — now includes stories, videos, and shopping
If we followed a very strict, step-by-step SDLC for Instagram, each phase could take months or years. Adding a new feature halfway through becomes incredibly difficult. One single SDLC approach doesn't fit every project.

Project Objectives Every Team Must Balance

  • Deliver the application faster to users
  • Incorporate user feedback
  • Ensure the application is high quality and free of errors

What are Software Development Models?

According to the specific needs of a project, the phases of the SDLC can be adjusted. The different ways of adjusting the phases of the SDLC are known as Software Development Models.

Software Development Models

Model 1 of 3 — Waterfall Model

1
Waterfall Model

A step-by-step approach where each phase completes before the next begins

It is a step-by-step way of developing software where each phase must be completed fully before moving to the next.

How it Works

Waterfall Model — sequential phase flow

When to Use

Best suited when:

  • Requirements are very clear and fixed
  • The project is small or simple
  • The customer won't change their mind often

Advantages & Disadvantages

Advantages

  • Simple to understand and use
  • Every phase has clear documentation — easy to track progress

Disadvantages

  • No feedback until the end
  • Hard to go back and fix something in earlier steps
  • If something is wrong, it's too late and costly to fix
  • Miscommunication
  • Slower time to reach users
  • Difficult to integrate new requirements or feedback

Model 2 of 3 — Agile Model, Scrum & Kanban

2
Agile Model

Build in small pieces through short, repeated cycles called sprints

Agile is a software development approach where the product is built in small pieces through short, repeated cycles called sprints (typically 1–4 weeks each).

Agile Model — iterative sprint cycles

Behind the Scenes — Two Key Teams

TeamResponsibility
Development TeamPlanning, designing, and building the software
Operations TeamImplementing and running the software once it's built

Agile Workflow

In Agile, we don't build the whole app at once. Instead, we break it into smaller pieces called features (e.g., the login screen, the add-to-cart button).

  • Each feature is built in short cycles
  • Different teams can work on many features at the same time
  • Developers pick up a feature, write the code, and mark it as done

The Delay Problem

Every time a new feature is ready, the Operations Team must manually handle servers, documents, system connections, and databases — creating a bottleneck that leads to delays, inconsistencies, and a higher chance of mistakes.

When to Use Agile

Best suited when:

  • Requirements are unclear or expected to change frequently
  • The client/stakeholder is available for regular feedback
  • The project is medium to large with evolving scope

Advantages & Disadvantages

Advantages

  • Reduces time to deliver the application
  • Reduces time to receive feedback
  • Flexible — easy to change plans if needed
  • Less risky — mistakes are caught early

Disadvantages

  • Miscommunication between Dev and Ops teams
  • Delay in Operations bottleneck

Model 3 of 3 — V-Model

3
V-Model

Verification and Validation — testing mapped to every development phase

V-Model stands for Verification and Validation Model. It is an extension of Waterfall — same sequential flow, but with a testing phase mapped to every development phase.

V-Model — verification and validation mapped at each phase

Verification vs Validation

Verification
"Are we building the product right?"
Checking that each phase is done correctly through reviews and walkthroughs — no code execution.
Validation
"Are we building the right product?"
Running actual tests on the code to confirm it meets user expectations.

When to Use

Best suited when:

  • Requirements are fixed and well-understood from the start
  • The project demands high quality and strict testing
  • The project is small to medium with a stable scope
