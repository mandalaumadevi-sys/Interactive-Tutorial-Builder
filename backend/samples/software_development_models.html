Why One SDLC Doesn't Fit All

  • Every software project differs in size, requirements, and flexibility
  • A simple to-do app has fixed requirements and takes weeks to build
  • Instagram started as photo sharing but now has stories, videos, reels, and shopping
  • A rigid step-by-step approach for Instagram could take years — adding features mid-way becomes incredibly difficult

Software Development Models

  • According to the specific needs of a project, the phases of the SDLC can be adjusted
  • The different ways of adjusting the SDLC phases are known as Software Development Models
  • Most commonly used models: Waterfall Model, Agile Model, and V-Model
  • Each model suits different types of projects based on size, flexibility, and requirements

Waterfall Model

Waterfall Model

  • A step-by-step way of developing software where each phase must be completed fully before moving to the next
  • Phases flow downward like a waterfall: Requirements → Design → Development → Testing → Deployment → Maintenance
  • No phase can begin until the previous one is fully signed off
  • The most traditional and oldest Software Development Model

When to Use Waterfall

  • Requirements are very clear and fixed from the beginning
  • The project is small or simple in scope
  • The customer will not change their mind often
  • Example: government systems, infrastructure software, or banking applications with strict compliance

Advantages of Waterfall Model

  • Simple to understand and use — straightforward sequential approach
  • Every phase has clear documentation, making it easy to track progress
  • Well-defined stages — team knows exactly what needs to be done at each step
  • Works well for projects where technology and requirements are well understood

Disadvantages of Waterfall Model

  • No feedback until the end — users only see the product after all phases are complete
  • Hard to go back and fix something in earlier steps once a phase is closed
  • If something is wrong, it is too late and costly to fix
  • Slower time to reach users — difficult to integrate new requirements or feedback mid-project

Agile Model

Agile Model

  • A software development approach where the product is built in small pieces through short repeated cycles called sprints
  • Each sprint typically lasts 1–4 weeks and delivers a working piece of the product
  • Different teams can work on many features simultaneously
  • Built to reduce time to deliver, incorporate feedback fast, and handle evolving requirements

Behind the Scenes — Two Key Teams

Development Team
  • 📐 Planning the features
  • 🎨 Designing the software
  • 💻 Building and coding
Operations Team
  • 🚀 Implementing the software
  • ⚙️ Running and managing servers
  • 📊 Monitoring performance

When to Use Agile

  • Requirements are unclear or expected to change frequently
  • The client or stakeholder is available for regular feedback
  • The project is medium to large with an evolving scope
  • Examples: consumer apps, startups, e-commerce platforms, social media products

Advantages of Agile Model

  • Reduces time to deliver the application — working features shipped every sprint
  • Reduces time to receive feedback — users interact with the product early
  • Flexible — easy to change plans based on feedback or market changes
  • Less risky — mistakes are caught early in short sprint cycles, not at the end

Disadvantages of Agile Model

  • Miscommunication between Dev and Ops teams — handoffs between teams can create gaps
  • Delay in Operations — every new feature requires manual server setup, document updates, and database management
  • These repeated manual tasks create a bottleneck that slows delivery
  • Requires highly disciplined teams and active stakeholder involvement to succeed

Scrum Framework

  • A structured framework where work is done in fixed-length cycles called sprints (1–4 weeks)
  • Each sprint delivers a working piece of the product
  • Product Owner decides what to build, prioritises the backlog
  • Scrum Master facilitates the process and removes blockers — Development Team (5–9 people) builds the product
Best For
Teams needing structure, clear deadlines, and regular deliverables

Kanban Framework

  • A visual workflow method where teams track tasks on a Kanban board
  • Works in any industry and can be customised to fit any team
  • Board has 3 columns: To Do → In Progress → Completed
  • Tasks move across the board as work progresses — no fixed sprint length required

V-Model

V-Model

  • V-Model stands for Verification and Validation Model
  • It is an extension of the Waterfall model — same sequential flow, but with a testing phase mapped to every development phase
  • For every development activity on the left side, there is a corresponding testing activity on the right side
  • The shape of the process forms a 'V' — development goes down, testing goes up

Verification vs Validation

Verification ✓
"Are we building the product right?"
  • 🔍 Checking each phase is done correctly
  • 📄 Reviews and walkthroughs
  • ❌ No code execution required
Validation ✅
"Are we building the right product?"
  • 🧪 Running actual tests on the code
  • 👤 Confirms it meets user expectations
  • ✅ Code execution required

When to Use V-Model

  • Requirements are fixed and well-understood from the start
  • The project demands high quality and strict testing
  • The project is small to medium in size with a stable scope
  • Examples: medical device software, aviation systems, safety-critical embedded systems

Key Takeaways

  • Software Development Models define how SDLC phases are arranged to fit different project needs
  • Waterfall — sequential, best for fixed requirements and simple projects
  • Agile — iterative sprints, best for evolving requirements and regular feedback
  • V-Model — testing mapped to every development phase, best for high-quality strict-testing projects
