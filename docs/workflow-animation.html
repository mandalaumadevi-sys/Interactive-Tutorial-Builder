The Interactive Tutorial Builder

Give it one course session — an HTML page or a PowerPoint deck — and a team of AI agents turns it into a single, self-contained interactive HTML tutorial: content split into guided steps, concept diagrams re-drawn as animations, every step gated by a quiz, and a final assessment — all checked at four levels before a human signs off.

The big picture

What it does — and why it matters

The one-sentence pitch for any audience: it converts passive reading material into an active, validated learning experience — automatically, but never blindly.

① One input, one artifact

A single session deck goes in. A single self-contained .html tutorial comes out — no build step, no dependencies, opens in any browser.

② A guided experience, not a cheat-sheet

Content is progressively revealed in blocks. Concept diagrams become step-by-step animations. Each block is locked behind a quiz, and the session ends with an assessment.

③ Quality is engineered in

Every agent grades its own work against a rubric and retries. A session-level judge scores the whole thing. Two human gates can redirect the flow. Nothing ships unchecked.

Built on LangGraph

A stateful graph of nodes with checkpointing (resumable mid-run), conditional routing, and two interrupt points for the humans. The graph always terminates.

Deterministic where it can be

Ingestion, assembly, and format conversion are plain code — fast, free, reliable. LLMs are reserved for the creative work: division, content, animation, questions, judging.

It gets smarter per course

Cross-session memory remembers concepts already taught, quiz topics used, and your feedback — so the next session in a course builds on the last instead of repeating it.

No AI

🔎 How images are found

Pure parsing. BeautifulSoup does find_all("img") (HTML) and python-pptx reads PICTURE shapes (PPTX). For each one we record an ImageRef: its src, alt text, caption (from the <figure>/figcaption), nearest heading, pixel size, and how many times the same image repeats across the document (a 20×-repeated logo is obviously decorative).

Vision

🎬 How an image reaches the model

The local image file is base64-encoded into a data: URL and attached as a standard OpenAI-style image_url message part. Agent 1 decides which images are concept-bearing; Agent 2 then receives the chosen image plus its alt/caption, "sees" it, and re-draws it as an animation — it never embeds the raw screenshot.

The cast

Five specialist agents (plus a judge)

Each agent has one job, its own prompt, and its own rubric. They are deliberately narrow — that is what makes each one easy to validate and easy to retry in isolation.

LLM

🧱 Block Divider

Groups the raw sections into 3–7 cohesive teaching blocks. It only decides which sections group together — it never rewrites content, so nothing is ever lost.

In: normalized sections + objectives → Out: a BlockDivision (titles, groupings, reasoning)
Rules (count 3–7, full coverage, balanced) + block_divider rubric → retry once.
LLM · vision

✍️ Agent 1 — Content & Layout

The per-block orchestrator. Rewrites one block into clean house-style HTML and decides, per image, whether it carries a concept worth animating. It is the only caller of Agent 2.

In: one block + images + memory → Out: learner-facing HTML with animations placed inline
Structural gate (real body text — catches empty blocks) + content rubric → retry.
LLM · vision

🎬 Agent 2 — Animation

Given one concept image, it re-draws the diagram natively as a self-contained, auto-looping step-by-step reveal (inline SVG/CSS/JS), matching a reference-animation style library.

In: one image + its alt/caption → Out: namespaced animation HTML fragment
Rules (self-contained, namespaced, reduced-motion, has <svg>) + visual rubric → retry.
LLM + parser

❓ Agent 3 — MCQs

Writes the per-block multiple-choice questions that gate progress. Runs the fixed MCQ prompt verbatim, then deterministically parses the -END- format into interactive QUIZ_DATA.

In: the same raw block (not Agent 1's HTML) → Out: 2–3 MCQs per block
MCQ rules (4 options, 1 correct, explanation, no leak) + mcq rubric → retry.
LLM

📝 Agent 4 — Final Assessment

Runs once after all blocks. Produces ~5 descriptive, open-ended questions with model answers spanning the whole session (tagged with Bloom's levels), avoiding topics already quizzed.

In: all built blocks + objectives → Out: descriptive Q&A carousel
Rules (answer present, right length, no padding) + assessment rubric → retry.
LLM · judge

⚖️ Quality Judge

Not a producer — a reviewer. Scores the whole assembled session on cross-block dimensions no single agent can judge about itself: learning flow, objective coverage, depth consistency, MCQ variety, assessment synthesis.

In: the full draft → Out: per-dimension scores → weighted avg vs 7.0 → pass/fail
Drives the bounded refine loop, then escalates to the human (HITL #2).

Orchestration detail (for engineers): per-block work fans out across a thread pool — N × (Agent 1 + Agent 3) run concurrently, with Agent 2 nested inside Agent 1 per image. State is stored as plain JSON dicts so it round-trips cleanly through the SQLite checkpointer.

The answer to "what are the levels?"

Four levels of quality control — defense in depth

Quality isn't one check at the end. It's four independent layers, cheapest first. A problem has to slip past all four to reach the learner — and the last layer is a human.

1

Deterministic rule checks  ·  free, no LLM

Fast structural guards on every agent's output: does the block have real body text (not just a heading)? Does the animation have an <svg>, namespacing, and a reduced-motion query? Do the MCQs have 4 options, exactly one correct, and an explanation? A failure triggers an immediate corrective re-run.

Where: tools/validation_tools.py, skills/*_validator.py · runs on every agent
2

Eval-set self-validation  ·  the agent grades itself against its rubric

Each agent's output is scored 0–10 on every dimension of its own eval-set rubric by an LLM judge, calibrated with good/bad examples. Below threshold (7.0) → the agent regenerates once with the reviewer's specific notes. Catches the qualitative problems rules can't see — thin explanations, weak distractors, low-fidelity animations.

Where: self_validate() + eval-sets/<agent>/ · runs inside each agent
3

Session-level judge + bounded self-refine  ·  the whole tutorial, scored

After draft assembly, one judge scores the entire session on cross-block dimensions. If it fails and retries remain, the system re-runs only the responsible stage (content / MCQ / assessment) with the judge's notes, then re-scores. Bounded: usually one retry, then it escalates — never an infinite loop.

Where: steps/final_quality_check.py + the quality → refine → quality edge
4

Human-in-the-loop gates  ·  the final authority

Gate #1 — Block review: you approve the proposed division or send feedback to re-divide. Gate #2 — Final review: you approve, edit, or reject the scored tutorial (reject loops all the way back to division). A block that failed its own self-checks is forced to this gate even if the overall score passed — nothing questionable ships silently.

Where: LangGraph interrupt_before → surfaced in the web UI
The spec, not an afterthought

Eval-sets — the rubrics that define "good"

Guiding principle of the build: "eval-sets are the spec." Each agent is built against its rubric, not against vibes. An eval-set is a small folder per agent containing three things — and the same files power both the self-validation (Level 2) and the final judge (Level 3).

📋 rubric.json

The weighted dimensions to score (e.g. clarity, accuracy, Bloom's level) and the pass threshold (7.0).

✅ good_examples.json

Reference outputs that should score high — calibrates the judge's top of the scale.

❌ bad_examples.json

Outputs that should score low — teaches the judge exactly what to penalize.

Are we using a golden dataset?

Golden examples & the offline judge eval

Honest answer: outputs here are open-ended HTML, animations, and prose — there's no single correct string to exact-match against. So instead of metric-based comparison, we keep human-authored, labelled golden exemplars and evaluate with an LLM-as-judge against a rubric. Those exemplars are used two ways.

at runtime

Calibration for the judge

Each agent's good_examples / bad_examples are injected into the self-validation and quality prompts so the judge scores calibrated against known-good and known-bad — not on a hunch.

offline

A golden-set regression eval

A harness (tutorial-builder eval) replays every labelled exemplar through the same judge and checks the labels hold: good should pass, bad should fail. It reports judge accuracy — a real, ground-truth-labelled number.

How the offline eval avoids cheating — leave-one-out

When an exemplar is scored, it is removed from the few-shot context the judge sees, so the judge can't recognise and rubber-stamp itself. Headline metric = (correctly-classified examples) ÷ (total), per agent and overall.

Yes — and it's bounded

Self-refinement: two loops, always terminating

"Bounded autonomy" is a core principle: the system improves its own work, but can never spin forever. There are two distinct self-correction loops, and both have hard limits.

Loop A · inside each agent

Self-validation retry

An agent scores its own output (Levels 1 & 2). If it's below the bar, it regenerates once with the specific fixes it was given — then keeps the better result. Local, cheap, immediate.

Limit: TB_SELF_VALIDATE_RETRIES (default 1)
Loop B · whole session

Quality-judge refine loop

If the session judge fails, the graph re-runs only the responsible stage with the judge's notes and re-scores. When retries are exhausted, it doesn't loop — it escalates to the human (HITL #2).

Limit: TB_MAX_REFINE_ATTEMPTS (default 1) → then escalate

The key safety property: the graph always terminates. Retries are capped, the refine loop falls through to a human, and a human reject loops back to division (also gated). There is no path to an infinite run.

For Q&A

Anticipated questions — crisp answers

Why agents instead of one big prompt?

Narrow agents are individually testable and individually retryable. When the MCQ rubric fails, we re-run just the MCQ agent — not the whole tutorial. Each agent also gets its own rubric and prompt, which keeps quality measurable rather than a single black box.

What stops it from hallucinating content?

Three things: the Block Divider only groups sections (it never invents prose, so source content can't be lost); Agent 2 is told to build animations only from what the image and its caption actually show; and Levels 2–3 score accuracy and material-alignment against rubrics, with a human as the final gate.

How expensive is a run?

Deterministic stages (ingest, assembly, parsing) are free. LLM cost scales with the number of blocks and images. The biggest multiplier is the refine loop — which is why it's bounded to one attempt by default. A cost ledger is kept per run.

What if the AI gets it wrong?

Two human gates. At block review you can redirect the structure; at final review you can approve, edit in place, or reject to regenerate from division. And any block that fails its own self-check is force-routed to the human even if the overall score passed.

Can I run it without spending on the API?

Yes — there's a built-in mock LLM (TB_LLM_MODE=mock) that runs the whole graph end-to-end on canned output, for testing the wiring at zero cost. Real runs use OpenRouter (Claude models by default).

What does "interactive" mean in the output?

One self-contained HTML file with a stepper UI: content reveals block by block, each block is locked behind its MCQs until answered, concept diagrams are live animations, and it ends with the assessment carousel — no server, no build, opens in any browser.
